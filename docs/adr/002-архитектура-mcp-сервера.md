# ADR-002: Всё-в-одном локальный MCP-сервер на TypeScript

**Статус**: Accepted
**Дата**: 2026-07-03
**Авторы**: @aleksandr, Claude

---

## Контекст

Пользователь выбрал форму инструмента «MCP-сервер для Claude» с полным автопилотом.
Канал откликов — браузер (ADR-001), значит процесс обязан жить там же, где браузерный
профиль, — на Mac пользователя; remote-вариант на Cloudflare Workers (как предыдущий
MCP-проект пользователя) невозможен. Открытым остался вопрос композиции и языка.

---

## Рассмотренные варианты

### Вариант A: всё-в-одном локальный MCP-сервер на TypeScript

Один Node-процесс: MCP (Streamable HTTP на localhost) + Playwright + node-cron + better-sqlite3.
Процесс постоянный (launchd), Claude Desktop подключается к нему по HTTP — stdio не подходит:
stdio-сервер спавнится самим Desktop, что привязало бы автопилот к открытому Desktop
и породило бы второй экземпляр процесса (конфликт за SQLite и браузерный профиль).

**Плюсы**:

- Один репозиторий, один процесс, один лог и дебаггер
- Официальный `@modelcontextprotocol/sdk` — самая зрелая MCP-экосистема
- Playwright на родной платформе
- Встроенный планировщик — автопилот без внешнего launchd
- У пользователя есть опыт MCP-серверов на TS (Cloudflare, OAuth 2.1, 22 инструмента)
- Контейнеризуется целиком при переезде на VPS

**Минусы**:

- Код замкнут на проект (митигируется: ядро — чистые функции)

### Вариант B: Python-ядро (CLI) + тонкая MCP-обёртка (FastMCP)

**Плюсы**:

- Python — основной рабочий язык пользователя
- CLI-ядро тестируется без MCP; куски легко утащить в другие Python-проекты

**Минусы**:

- Два слоя и интеграция между ними; внешний планировщик (launchd) — лишняя точка отказа
- MCP-экосистема Python у пользователя не обкатана

### Вариант C: ядро на VPS + remote MCP

**Плюсы**: всегда онлайн, Mac не нужен.
**Минусы**: hh.ru с датацентровых IP чаще ловит капчи — повышенный риск для аккаунта; заметно сложнее. Отклонён как старт, возможен как эволюция.

---

## Решение

**Выбран вариант A: всё-в-одном локальный MCP-сервер на TypeScript.**

---

## Обоснование

Для полного автопилота критично минимальное число движущихся частей — встроенный
планировщик A выигрывает у связки «CLI + launchd» варианта B. Опыт пользователя в TS
MCP снимает главный аргумент за Python. Требование переиспользования закрыто
архитектурно: `pipeline/` — чистые функции без зависимостей на браузер и MCP,
при необходимости портируются (решение пользователя: «если что, код портируем»).

---

## Последствия

**Положительные**:

- Быстрый старт по знакомому шаблону
- Автопилот работает из коробки после `npm start`

**Отрицательные / риски**:

- Mac должен быть включён в окна расписания — пропущенная сессия просто не выполняется (следующая доберёт лимит)
- При желании «всегда онлайн» — отдельный проект переезда (вариант C, v1.1+)

**Действия**:

- [ ] `pipeline/`, `filters`, `llm/*` — без импортов из `browser/` и `mcp/` (проверка на ревью)
- [ ] Автозапуск процесса при логине (launchd plist — один раз при установке)
- [ ] Пересмотреть при переезде на VPS или при портировании ядра
