# ADR-001: Канал откликов — браузерная автоматизация вместо официального API

**Статус**: Accepted
**Дата**: 2026-07-03
**Авторы**: @aleksandr, Claude

---

## Контекст

Агенту нужно программно искать вакансии и отправлять отклики на hh.ru. Проверено 2026-07-03:

- Анонимные запросы к `api.hh.ru/vacancies` возвращают `403 forbidden` — публичный доступ закрыт
- Отклики через API (`POST /negotiations`) требуют OAuth-приложения, зарегистрированного на dev.hh.ru; заявки модерируются вручную до 15 календарных дней
- Залогиненная браузерная сессия пользователя работает: поиск и кнопка «Откликнуться» доступны

Пользователь активно ищет работу — ждать модерацию 15 дней означает потерять
2 недели откликов. Подробности: `Personal/.wiki/patterns/hh-api-access.md`.

---

## Рассмотренные варианты

### Вариант 1: Официальный API hh.ru (OAuth)

**Плюсы**:

- Легально и стабильно: контракт API, никакой зависимости от вёрстки
- Структурированные данные вакансий, надёжный `POST /negotiations`
- Работает headless на любом сервере

**Минусы**:

- Старт через 1–15 дней (модерация заявки)
- Регистрация приложения, поддержка OAuth-флоу и refresh-токенов
- Заявку на «автоматизацию откликов» могут не одобрить

### Вариант 2: Браузерная автоматизация (Playwright, залогиненная сессия)

**Плюсы**:

- Работает сегодня, без регистраций и ожиданий
- Видит то же, что пользователь (включая hh PRO функции)
- Сессия в persistent-профиле живёт долго

**Минусы**:

- Серая зона правил hh.ru; риск капчи и (теоретически) санкций к аккаунту
- Хрупкость: смена вёрстки ломает селекторы
- Требует включённого Mac с браузером

**Риск-митигация**: лимит 10 откликов/день (лимит hh — 200), паузы 3–7 мин,
человекоподобные задержки, немедленный стоп при капче (не решаем принципиально),
все селекторы изолированы в модуле `browser/`.

### Вариант 3: Гибрид — старт на браузере, параллельная заявка на dev.hh.ru

**Плюсы**: старт сейчас + легальный канал потом.
**Минусы**: двойная работа по поддержке двух каналов; пользователь отложил регистрацию приложения.

---

## Решение

**Выбран вариант 2: браузерная автоматизация.**

---

## Обоснование

Ключевой критерий — время до первого отклика: вариант 2 даёт его сегодня, вариант 1 —
через недели с риском отказа в модерации. Вариант 3 отклонён как преждевременный:
пайплайн абстрагирован от канала (этап `apply` — отдельная функция), поэтому переезд
на API при необходимости не потребует переписывания ядра — гибрид можно включить позже
без двойной работы сейчас.

---

## Последствия

**Положительные**:

- Агент работает с первого дня
- Полный доступ ко всему, что видит залогиненный пользователь

**Отрицательные / риски**:

- Зависимость от вёрстки hh.ru — закладываем HTML-фикстуры и быструю починку селекторов
- Риск капч при разгоне темпа — темп зафиксирован консервативно
- Автопилот привязан к включённому Mac

**Действия**:

- [ ] Селекторы hh.ru — только в `browser/`, с фикстурами для тестов
- [ ] Детект капчи и разлогина с первым релизом (FR-012)
- [ ] Пересмотреть решение: при первой капче/блокировке ИЛИ если появится время на регистрацию приложения dev.hh.ru
