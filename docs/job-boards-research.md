# Рисерч: российские площадки вакансий без логина

Дата: 2026-07-13. Все выводы основаны на реальных запросах (curl/HTTP) с российского IP,
без логинов и обхода защит. Цель — дополнительные источники вакансий для пайплайна
(сейчас — только hh.ru через Playwright, `src/browser/hh.ts`).

## Детали по площадкам

### hirehi.ru — приоритет 1
- `GET https://hirehi.ru/api/search/jobs?query=python` → JSON, `total_count: 16089`, пагинация `page=` (27/стр), фильтры (удалёнка: 8 193, senior: 4 977).
- `GET https://hirehi.ru/api/jobs/{id}` → полная карточка: `description`, `requirements`, `tasks_details`, `conditions_details`, `salary_display`, `format`, `level`, `skills_list`.
- Сам является агрегатором (телеграм-каналы + рекрутеры) → **нужна дедупликация с hh.ru**; механика отклика — отдельный вопрос (прямой ссылки на источник в карточке нет).
- API неофициальный.

### Habr Career — приоритет 2
- Список: `GET https://career.habr.com/api/frontend/vacancies?q=python&page=1` → `{list, meta:{totalResults, totalPages}}`. Фильтры: `remote=true`, `salary=`, `qid=` (3/4/5 = Jr/Mid/Sr), `s[]=` специализация, `per_page` ≤ 50, `sort=date`.
- Карточка: `GET https://career.habr.com/vacancies/{id}` — статический HTML, описание в `div.vacancy-description__text` + JSON-LD JobPosting.
- RSS: `/vacancies/rss?q=...`. Антибота нет, UA не проверяется, 10 подряд запросов — ок.
- Поля списка: salary{from,to}, qualification, skills, remoteWork, company{rating, accredited}, locations.

### getmatch.ru — приоритет 3
- `GET https://getmatch.ru/api/offers?limit=100&offset=0` → `{meta:{total:720}, offers:[...]}`, `description_html` прямо в списке, вилки net/gross, `english_level`, `location_items` (remote).
- Текстового поиска в API нет — выкачивать всё (8 запросов) и фильтровать локально.
- На веб-страницах есть Яндекс SmartCaptcha, но API отвечает голому curl.

### SuperJob
- Официальный API `api.superjob.ru/2.0/vacancies/`: нужен только бесплатный app key (регистрация на api.superjob.ru/register — делает пользователь), заголовок `X-Api-App-Id`, **OAuth пользователя не нужен**. Лимиты: 120 req/min, ≤500 сущностей на запрос.
- Fallback: HTML отдаётся без логина, все данные в `window.APP_STATE` (entity-store) + JSON-LD на карточке. За Qrator; замечен «децой» на первом холодном запросе (отдал чужую страницу) → держать cookies и валидировать ответ.
- IT-сегмент скромный: «python» Москва — 51 вакансия. Скорее дополнение.

### Работа России (trudvsem.ru)
- Официальное открытое API: `GET https://opendata.trudvsem.ru/api/v1/vacancies?text=...`, региональное: `/vacancies/region/77?text=...`. Без ключа. `limit` ≤ 100, `offset` = **номер страницы** (0-based), `modifiedFrom=` для инкрементального пуллинга.
- Поля: duty/requirements (plain text, медиана >1100 симв.), зарплата у ~90%, **email и телефон работодателя прямо в API**, координаты, `vac_url` (карточка открывается без логина; отклик на портале — через ЕСИА).
- Минусы: объём по IT мал (python Мск — 125), API медленное (~4.5 с/запрос).

### Rabota.ru
- HTML SSR без логина: `https://www.rabota.ru/vacancy?query=python` → 200, счётчик «2311 вакансий», 13 JSON-LD JobPosting на 1-й странице; карточка `/vacancy/{id}/` — полная SSR.
- **Но**: страницы 2+ — пустой shell (infinite-scroll на JS), а внутренний `api.rabota.ru/v5/vacancies/search.json` анонимно игнорирует keyword. Итого полноценный сбор — только через Playwright.
- Антибот Qrator: 403 при частых запросах/куцем UA; с полным UA и паузами — ок. Город — через поддомены.
- Официальный API есть (dev.rabota.ru), но OAuth + регистрация приложения под логином, партнёрская модель.

### geekjob.ru / budu.jobs
- geekjob: `GET https://geekjob.ru/json/find/vacancy?qs=python&page=1` (поиск работает) + JSON-LD в карточке. Но всего ~200 вакансий.
- budu.jobs: ~70 вакансий, много не-IT, API нет. Не стоит усилий.

### Zarplata.ru — не добавлять
- С ~2023 полностью на платформе hh.ru: общие ID вакансий, те же data-qa селекторы, база — строгое подмножество hh (~16% объёма; python Мск: 215 vs 3 968). Каждая проверенная вакансия открывается на hh.ru по тому же ID.
- `api.zarplata.ru` жив и зеркалит api.hh.ru: `/vacancies` → 403 анонимно (как и у hh с 2024 — нужен app-token с dev.hh.ru), `/dictionaries` → 200.

### Avito Работа — сложно, отложить
- Публичного read-API нет (Job API — только управление своими объявлениями работодателя).
- Первый запрос выдачи → 200 с полными 50 карточками (данные SSR в HTML, `data-marker` стабильны), но дальше — **тихий бан**: HTTP 200 с пустой оболочкой. Карточки по прямым URL устойчивее.
- Рабочая схема по опыту 2024–2026: ротируемые мобильные прокси + anti-detect + низкая частота. Дорого в поддержке, ToS-риск выше, чем у hh.

## Рекомендация

Первая волна (всё — голый HTTP, без Playwright, дёшево и стабильно):
1. **hirehi.ru** — максимальный ROI, но обязательна дедупликация (по названию+компании) с hh.ru.
2. **Habr Career** — IT-only, грейды и стеки тегами, удобно для скоринга.
3. **getmatch.ru** — мало, но качественно (честные вилки).

Вторая волна: **SuperJob** (после регистрации app key пользователем), **trudvsem** (дешёвое дополнение).
Rabota.ru — только если готовы расширять Playwright-слой. Avito и Zarplata — не брать.

## Направление: прямые письма HR (email-flow)

Цель следующего этапа — не отклик через площадку, а поиск HR-почты и письмо напрямую.
Принятые решения (2026-07-13):

- Почта — атрибут **компании**, кэшируется по аналогии с company research (включая негативный результат `not_found` с TTL).
- Активного «дешёвого поиска» почты нет — до скоринга только чтение уже известного:
  - при инжесте вакансии почта из payload источника (trudvsem отдаёт в API) бесплатно складывается в кэш;
  - перед скорингом — проверка кэша: свежий `not_found` → вакансию не скорим (письмо некуда слать).
  - Скрейп сайта компании (mailto/контакты) сознательно НЕ делаем: дорогой поиск и так случается
    единицы раз в день (только после скоринга, с кэшем 30 дней) — экономить нечего. Вернуться к идее,
    только если объём дорогих вызовов станет заметным в деньгах.
- Инструмент поиска: **Perplexity (sonar)**, отдельным вызовом со своим узким промптом.
  Экономика по факту (llm_calls): скоринг ≈ $0.029/вакансия, sonar-вызов ≈ $0.006/компания (с request-фи) —
  поиск почты в ~5 раз дешевле скоринга и кэшируется по компании ⇒ выполняется **до скоринга** как гейт.
  Hunter/Snov отложены: $34/мес Starter, сомнительное покрытие российских доменов; идея на потом —
  бесплатные верификации Hunter (50/мес) для проверки найденных адресов перед отправкой.
- **Не объединять** поиск почты с company research в один Perplexity-вызов: проверено отдельно,
  скоуп размывается и деградируют оба результата.
- Порядок email-flow: поиск по источникам → хард-фильтры → почта через Perplexity (кэш по компании;
  свежий `not_found` → скип) → скоринг → research → письмо → **очередь с ручным подтверждением отправки**.
- Скоуп: email-flow — **только новые площадки** (hirehi, Habr Career, getmatch, trudvsem);
  hh.ru продолжает работать как сейчас, через браузерный отклик.
- Отправка: полного автомата нет — письма копятся в очереди (draft), Alex просматривает и подтверждает;
  автоматизация отправки возможна позже, когда качество писем подтвердится.
- Первая волна источников: hirehi.ru, Habr Career, getmatch.ru, trudvsem.ru (у trudvsem почта прямо в API).
- Сильные вакансии без найденной почты не выбрасывать, а складывать в очередь «ручной отклик через площадку».

Общие замечания:
- Все неофициальные JSON API могут поменяться без предупреждения — закладывать валидацию схемы ответа и алерт при расхождении.
- Отклик («apply») на новых площадках — отдельная задача; этот рисерч покрывает только чтение вакансий. Для начала новые источники могут работать в режиме «скоринг + ссылка на ручной отклик».
- ToS большинства площадок формально запрещают автоматический сбор (как и hh.ru) — риск-профиль тот же, что у текущего пайплайна; вежливый рейт (~1 req/s) обязателен.
